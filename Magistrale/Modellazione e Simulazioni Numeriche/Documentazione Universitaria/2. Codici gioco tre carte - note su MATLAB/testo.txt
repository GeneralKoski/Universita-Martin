Abbiamo iniziato a commentare qualcosa sulla struttura e la filosofia di Matlab, in primis sottolineando uno degli aspetti più importanti dell'ambiente:

Matlab è un linguaggio interpretato.
Fedeli al nostro approccio pragmatico/sperimentale (una lingua si impara davvero parlandola), rimandiamo (come sempre faremo nel seguito) al DIARIO della nostra sessione di lavoro. Più in dettaglio, abbiamo avuto il pretesto per imparare un certo numero di cose riguardo alle funzioni in Matlab, fra le altre:

Le funzioni sono definite entro file. Non ci vuole molto a notare come questi file hanno lo stesso nome delle funzioni che in essi sono definite; l'estensione del file deve essere .m
Osservando cosa Matlab riconosca come code files (cfr quanto generato dal comando what), abbiamo osservato come un file sorgente (code file, appunto) di Matlab può contenere funzioni o script file. Ci torneremo sopra, ma credo sappiate cosa aspettarvi dai secondi ... in breve: la esecuzione di uno script corrisponde in tutto e per tutto a digitare quegli stessi comandi nella finestra di comando. Questo porta con sé lo stesso ambito di validità che una variabile avrebbe in questo contesto (entra a far parte dello spazio di lavoro...). Anche su questo torneremo a riflettere: le regole di scope per le variabili in una funzione sono un po' quelle di sempre (una variabile ha l'ambito di validità della funzione entro cui è definita).
È possibile scrivere un help per la funzione: si tratta del primo commento (ricordate: è un commento un testo che segue il carattere %) che normalmente segue la riga che definisce la sintassi per la funzione.
Lo ripeteremo più e più volte: la prima riga effettiva (non un commento...) che scriviamo per una funzione è sempre della forma argomenti_in_output = nome_funzione(argomenti_in_input). Ricordate? È possibile passare un numero di argomenti <= a quelli che troviamo in tale prima riga, con la avvertenza extra che si può sfruttare una variabile varargin che consente di radunare un numero variabile di argomenti (utile in particolare quando una funzione deve ammettere come argomento un'altra funzione, di cui non possiamo conoscere a priori il numero di argomenti). Ricordate a proposito della ultima osservazione (varargin): ne abbiamo parlato a proposito della funzione (quad) che calcola integrali definiti. Essa prende come primo argomento una funzione, il cui primo argomento deve essere la variabile rispetto a cui integriamo. Un numero variabile di argomenti in input ci consente di passare argomenti supplementari a questa funzione (in soldoni: quad non può sapere a priori quanti argomenti prende la funzione di cui deve calcolare l'integrale, oltre alla variabile sulla cui dipendenza funzionale dobbiamo integrare, che - lo ricordo! - deve sempre essere la prima nella lista degli argomenti).
 Sfruttando nargin (numero di argomenti in input) abbiamo scritto un valore di default per un argomento in input. Ricordate che esiste anche nargout (numero di argomenti in output).
Particolarmente generoso si dimostra Matlab nel farci visualizzare il file sorgente della grande maggioranza delle funzioni predefinite (abbiamo fatto l'esempio digitando il comando type quad.m, visualizzando il codice sorgente della funzione quad, che è una semplice funzione di quadratura, ovvero di calcolo di integrali definiti). Scopriamo peraltro che la maggior parte delle funzioni predefinite sono scritte per essere mandate in esecuzione in modalità interpretata (tanto quanto quelle che scriviamo noi).
Abbiamo scoperto, provando a digitare type sin.m, che ci sono però funzioni (ed il seno è una di queste) che sono disponibili compilate (viene mandato in esecuzione codice compilato, efficiente ed ottimizzato).
