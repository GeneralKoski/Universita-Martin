# 2.3 - `solve_excluding_python` (solve-and-block)

**Cosa fa:** il *secondo* metodo di unicità. Trova una soluzione **diversa** da una data. È il gemello Python del modello MiniZinc `sudoku_non_unique_check.mzn`.

## Il codice

```python
def solve_excluding_python(grid, forbidden):
    work = copy_grid(grid)

    def backtrack():
        cell = find_empty(work)
        if cell is None:                      # griglia completa
            if any(work[r][c] != forbidden[r][c]
                   for r in range(9) for c in range(9)):
                return True                   # diversa da forbidden → OK
            return False                      # uguale a forbidden → rifiuta
        row, col = cell
        for value in candidate_values(work, row, col):
            work[row][col] = value
            if backtrack():
                return True
            work[row][col] = 0
        return False

    return work if backtrack() else None
```

## L'idea

```
forbidden = S₁  (la prima soluzione, già trovata altrove)

cerca una griglia valida QUALSIASI, purché diversa da S₁ in ≥1 cella

trova una griglia diversa?
   SÌ  → restituisce S₂        → esiste una seconda soluzione → MULTIPLE
   NO  → restituisce None      → S₁ è l'unica                → UNIQUE
```

## Il filtro a fine ricerca

La differenza con `solve_python` normale è **una sola riga**, al momento in cui la griglia è completa:

```python
if any(work[r][c] != forbidden[r][c] for r,c): return True   # diversa → accetta
return False                                                  # identica → rifiuta, continua
```

`any(work[r][c] != forbidden[r][c] ...)` è l'equivalente Python esatto di:

```minizinc
exists(i,j) (grid[i,j] != known_solution[i,j])
```

cioè il **blocking constraint** del modello MiniZinc.

## Counting (2.2) vs Solve-and-block (2.3)

| | Counting | Solve-and-block |
|---|---|---|
| Come | conta soluzioni fino a 2 | trova S₁, poi cerca S₂ ≠ S₁ |
| Verdetto unico | found=1 + albero esaurito | seconda ricerca → None (UNSAT) |
| Verdetto multiplo | found=2 | seconda ricerca → griglia |
| Costo su puzzle unico | esaustivo (esplora tutto) | esaustivo (la 2ª ricerca deve esaurire) |
| Velocità misurata | ~1.8× più veloce | più lento |

Entrambi sono richiesti dalla specifica perché il punto è **confrontarli**.

## Differenza implementativa con MiniZinc

Domanda: *"perché filtrare a fine ricerca invece di propagare la disuguaglianza?"*

> *"Per semplicità implementativa. In CP propagare `grid != forbidden` come blocking constraint taglierebbe l'albero prima ed è più efficiente - ed è infatti quello che fa il modello MiniZinc. In Python procedurale aggiungere logica di propagazione per un singolo vincolo non vale la pena: filtro solo a soluzione completa."*

## Frase secca per l'orale

> *"Solve-and-block: trovo una prima soluzione S₁, poi cerco una soluzione che differisca da S₁ in almeno una cella. È il gemello Python del modello `sudoku_non_unique_check.mzn`: `any(work != forbidden)` corrisponde a `exists(i,j) grid[i,j] != known_solution[i,j]`. Se la seconda ricerca fallisce (None), S₁ è unica; se trova qualcosa, ci sono ≥2 soluzioni."*
