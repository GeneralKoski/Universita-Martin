# 2.2 - `count_solutions_python` (counting con limite)

**Cosa fa:** controlla l'unicità di un puzzle **contando** le soluzioni, ma fermandosi a 2.

## Gli helper che usa

### `candidate_values(grid, row, col)` - forward checking
Risponde a: *"quali valori sono ancora legali per questa cella?"*

```python
used = set(grid[row])                          # valori già nella RIGA
used.update(grid[r][col] for r in range(9))    # valori già nella COLONNA
br = (row // 3) * 3; bc = (col // 3) * 3        # angolo del blocco 3×3
for r in range(br, br+3):
    for c in range(bc, bc+3):
        used.add(grid[r][c])                   # valori già nel BLOCCO
return [v for v in range(1,10) if v not in used]   # complemento in {1..9}
```

È la versione procedurale della propagazione di `alldifferent` su una singola cella.

### `find_empty(grid)` - `first_fail` scritto a mano
Sceglie **quale** cella vuota riempire: quella col dominio più piccolo.

```
0 candidati  →  ritorna subito  →  early fail (il backtrack fallirà subito)
1 candidato  →  ritorna subito  →  scelta forzata
altrimenti   →  tieni la più piccola vista finora
```

## Il codice

```python
def count_solutions_python(grid, limit=2):
    work = copy_grid(grid)
    found = 0
    first_solution = None

    def backtrack():
        nonlocal found, first_solution
        if found >= limit:          # short-circuit: ho già 2, basta
            return
        cell = find_empty(work)
        if cell is None:            # griglia piena = una soluzione completa
            found += 1
            if first_solution is None:
                first_solution = copy_grid(work)
            return
        row, col = cell
        for value in candidate_values(work, row, col):
            work[row][col] = value      # 1. PROVA
            backtrack()                 # 2. RICORRI
            work[row][col] = 0          # 3. UNDO
            if found >= limit:          # appena trovo la 2ª, taglio
                return
    backtrack()
    return found, first_solution
```

## Differenza con un solver normale

```
solver normale:   trova 1 soluzione → STOP
count_solutions:  trova 1 → continua → trova 2 → STOP (per il limit)
```

## Il trucco del `limit=2` (LA cosa da capire)

```
NON serve il numero esatto di soluzioni  →  sarebbe #P-completo
serve solo:  "sono ≥ 2 o no?"            →  basta contare fino a 2
```

Lo short-circuit `if found >= limit: return` taglia l'esplorazione appena trovata la seconda → velocissimo sui puzzle non-unici.

## Il verdetto

```
found = 0  →  unsat    (nessuna soluzione)
found = 1  →  unique   (e l'albero è stato esaurito senza altre)
found = 2  →  multiple
```

## Aggancio teorico (CRUCIALE)

> Contare TUTTE le soluzioni di un Sudoku è **#P-completo**. Ma non serve il conteggio esatto: basta sapere se ce n'è più di una. Limitando il contatore a 2, il problema diventa `#≥2`, che sta in **NP** (basta esibire 2 soluzioni come testimonianza). Quindi `limit=2` abbassa la classe di complessità da #P-completo a NP.

## Il pattern backtracking

```python
work[row][col] = value   # 1. PROVA un valore
backtrack()              # 2. RICORRI in profondità
work[row][col] = 0       # 3. ANNULLA (rimetti vuoto)
```

Modifica in-place → esplora → ripristina. Senza l'undo, lo stato resterebbe sporco per il ramo successivo.

## Domande probabili

- *"È #P-completo?"* → No, il limite a 2 lo rende NP.
- *"Per un puzzle unico come si comporta?"* → Esaustivo: deve esplorare tutto l'albero per dimostrare che la 2ª soluzione non esiste.

## Frase secca per l'orale

> *"Stesso solver backtracking, ma non si ferma alla prima soluzione: conta fino a 2. Il limite a 2 è il punto chiave per l'unicità - non mi interessa il conteggio esatto (#P-completo), solo se ci sono ≥2 soluzioni, che è in NP. found=1 + albero esaurito → unico; found=2 → multiple."*
