# 2.6 - `generate_puzzle` (loop di generazione)

**Cosa fa:** il **cuore** della pipeline. Trasforma una griglia completa in un puzzle minimale. Greedy con rollback.

## Il codice (parti che contano)

```python
def generate_puzzle(source_grid, backend, project_root, strategy, timeout_seconds, seed, method="counting"):
    rng = random.Random(seed)
    puzzle = copy_grid(source_grid)
    removal_log = []
    positions = iter_positions(strategy, rng)      # ← ordine dalla 2.5

    index = 0
    while index < len(positions):
        row, col = positions[index]
        if puzzle[row][col] == 0:                  # già rimossa → salta
            index += 1
            continue
        previous = puzzle[row][col]                # salva per il rollback

        # gestione del mirror (solo strategia symmetry)
        mirror = None; mirror_prev = None
        if strategy == "symmetry":
            candidate = (8 - row, 8 - col)
            if candidate != (row, col):
                mirror = candidate
                mirror_prev = puzzle[mirror[0]][mirror[1]]

        puzzle[row][col] = 0                       # PROVA a rimuovere
        if mirror is not None:
            puzzle[mirror[0]][mirror[1]] = 0       # rimuovi anche il mirror

        status = evaluate_puzzle(project_root, puzzle, backend, timeout_seconds, method=method)
        accepted = status["status"] == "unique"    # ← chiede a 2.2 o 2.3

        if not accepted:                           # ROLLBACK
            puzzle[row][col] = previous
            if mirror is not None:
                puzzle[mirror[0]][mirror[1]] = mirror_prev

        removal_log.append({...})
        index += 1

    clues = sum(1 for row in puzzle for v in row if v != 0)
    return {"puzzle": puzzle, "solution": source_grid, "clues": clues, ...}
```

## Lo schema (greedy con rollback)

```
parti da una griglia completa
per ogni cella, nell'ordine della strategia:
    salva il valore
    rimuovila (+ il mirror se symmetry)
    il puzzle è ancora UNICO?     ← chiama 2.2 (counting) o 2.3 (solve-and-block)
        SÌ → conferma la rimozione (cella resta vuota)
        NO → rollback (rimetti il valore, e il mirror)
→ puzzle finale minimale
```

## Perché backup + rollback

La rimozione modifica `puzzle` **in-place**. Se la rimozione non è accettata, devi ripristinare lo stato precedente prima di passare alla cella successiva. Per questo si salva `previous` (e `mirror_prev`).

## Il mirror in symmetry

```
strategia symmetry → ogni decisione rimuove DUE celle: (r,c) e (8-r, 8-c)
un rifiuto → blocca entrambe → "double tax"
→ ecco perché symmetry finisce con più clue
```

## Output

- `puzzle`: la griglia finale con i buchi
- `solution`: la griglia completa di partenza
- `clues`: numero di celle non-zero rimaste
- `removal_log`: tutti i tentativi (accettati e no), con tempi → serve per i plot

## Domande probabili (IMPORTANTI)

### *"È un greedy. È ottimo?"*
> *"No. È greedy locale: ogni cella o è rimossa o no, mai riconsiderata. Per arrivare al limite teorico (17 clue, McGuire 2012) servirebbe **backtracking sulla rimozione**: dopo un rifiuto, tornare indietro e provare a rimuovere altre celle in ordine diverso. Il mio greedy si ferma intorno a 22-25 clue. È il limite discusso nel report §7."*

### *"Perché 17 è il minimo teorico?"*
> *"McGuire, Tugemann, Civario (2012) hanno dimostrato con ricerca esaustiva assistita da computer che non esistono Sudoku validi e unici con 16 clue. 17 è il minimo. Il mio generatore non ci arriva perché è greedy, non esaustivo."*

### *"Differenza solving vs generation?"*
> *"Il solving è un CSP: dato un puzzle, trova la soluzione. La generation NON è un CSP - è una procedura di ricerca che USA il CSP come subroutine. `generate_puzzle` è un loop imperativo che chiama il check di unicità (CP) dopo ogni rimozione. Risolvere è CP; generare è un algoritmo costruito sopra il CP."*

## Frase secca per l'orale

> *"È il motore di generazione: parto da una griglia completa e rimuovo celle una a una nell'ordine deciso dalla strategia; dopo ogni rimozione controllo l'unicità con counting o solve-and-block, e se si perde l'unicità faccio rollback. È greedy con rollback locale - non ottimo: per il minimo teorico di 17 clue servirebbe backtracking sulle rimozioni."*
